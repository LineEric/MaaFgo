## 1. Context

`agent/mission_solver/` 已能用 ILP 求出 `{quest_id: count}` 最优刷本方案，但当前只是 CLI 工具，没有任何 pipeline 接入它。本功能把求解结果接到现有的"日常战斗状态机"，实现**一键周常自动清完**。

### 用户决策（已确认）

- **队伍策略**：用户指定一个固定 BBC team_config，所有 quest 复用
- **执行模式**：求解后直接逐关自动打（中途不弹用户确认）
- **入口**：新建顶级 task `周常任务自动执行`（不影响现有"日常战斗"）
- **数据获取**：选**方案 Y**（扩展数据更新工具自动拉中文名），失败回退**方案 X**（手工建表）

## 2. 现有"日常战斗"调用链（关键参考）

`assets/resource/base/pipeline/日常战斗.json` 是个 **JumpBack 状态机**：

```
日常战斗 (entry)
  → 通用战斗调度 (recognition=DirectHit, 多分支按"当前界面识别"决定下一步)
       ├─ [JumpBack]执行回主界面    → 回主界面 子图
       ├─ [JumpBack]执行队伍选择    → 队伍选择 子图（按 team_index 选 1–15 队）
       ├─ [JumpBack]执行章节导航    → 章节导航 → {冬木|奥尔良|...|梅塔特洛尼俄斯}
       │       → 进入主线记录 + [JumpBack]主线记录导航
       │       → 调用 "地图坐标导航" 节点 (general_navigation custom action)
       │       → 关卡归位 → [JumpBack]关卡滑动 → 关卡选择 (FeatureMatch + Click)
       ├─ [JumpBack]执行bbc战斗     → bbc战斗 → 执行BBC任务 (BBC 内部刷 N 次)
       ├─ [JumpBack]执行战斗结束    → 结束战斗 子图（处理弹窗、撤退、好友申请等）
       └─ 战斗完成信息              → 兜底退出（无任何 JumpBack 匹配时进入）
```

### JumpBack 语义复习

`[JumpBack]A` 候选：当前节点的 next 命中并跑完 A（及其子图）后，**跳回当前节点的 next 列表起点重新识别**，而不是结束。配合 `recognition=DirectHit` 永远命中，形成 "while True: dispatch on observation" 状态机。

### 章节切换的 pipeline_override 形态

`assets/options/chapter.json` 切章节时一次 override 多处：

```json
{
  "章节导航": { "next": "冬木" },
  "地图坐标导航": { "attach": { "chapter": "冬木" } },
  "找到进入主线记录章节": { "template": ["OoTTemple/冬木.png"] },
  "进入主线记录": { "template": ["OoTTemple/主线记录1-1.png"] },
  "点击主线记录": { "template": ["OoTTemple/主线记录1.png"] }
}
```

**切章节不是单纯改字符串，还要换多个 template 路径**。

具体关卡通过 `assets/options/quests/{章节}.json` 切换，覆盖 `地图坐标导航.attach.quests` 字段（坐标 key 名）。

## 3. **核心数据缺口**（实施前必须解决）

### 三套命名体系完全不交叉

| 系统 | 字段 | 巴比伦尼亚某关示例 |
|---|---|---|
| Atlas Academy / 求解器 | `quest_id` + `name`（**日文**） | `94032001` + `"廃都バビロン"` |
| 国服游戏官方中文 | (无字段) | `"遗失之都"` |
| MaaFgo 大地图坐标 | `q_name` in `map_coordinates.json` | `"废都巴比伦"` |

求解器的 `quest_enemies_CN.json` 实际 `name` 是日文（如 `quest_id=93000001` → `"屋敷跡"`），即使 `update_quest_data.py:148` 注释说"国服需要中文名"。

### `general_navigation` 严格要求

`general_navigation_action.py:80-121` 的 `attach.quests` 必须**严格 `==`** `map_coordinates.json` 中该章节下的某个 q_name（如 `"废都巴比伦"`）。

### 要让 quest_id 能自动导航的最小映射

```
quest_id → (chapter_cn, map_quest_key)
  例：94032001 → ("巴比伦尼亚", "废都巴比伦")
```

无法自动推导（日文 → 中文坐标 key 是意译，机器算不出）。

### 复用现有 `quests/*.json` 减少手工量

`assets/options/quests/{章节}.json` 已包含**所有 MaaFgo 当前支持自动导航的 free quest** + 它们的中文官方名（case `name`）+ 坐标 key（`pipeline_override.地图坐标导航.attach.quests`）。

所以缺口实际是：

```
quest_id (Atlas 数字) → 中文官方名（如 "遗失之都"）
```

然后通过 `quests/{章节}.json` 自动查到 `(chapter_cn, map_quest_key)`。

#### 5.1 新增 `agent/mission_solver/quest_index.py`

```python
"""
quest_id → (chapter_cn, map_quest_key) 索引构建器
"""

def build_quest_id_index() -> dict[int, tuple[str, str]]:
    """
    返回 {quest_id: (chapter_cn, map_quest_key)} 字典
    
    工作流程:
      1. 加载 war_id_to_chapter.json
      2. 加载 quest_enemies_CN.json (求解器数据)
      3. 加载 quest_id_to_zhcn.json (中文翻译表，方案 Y.1/Y.2 拉到的或方案 X 手工的)
      4. 加载所有 assets/options/quests/{章节}.json
      5. 对每个 quest_id:
          chapter = war_id_to_chapter[quest.warId]
          zhcn_name = quest_id_to_zhcn[quest_id]
          map_quest_key = quests/{chapter}.json 中找 case.name == zhcn_name 的 case 的 pipeline_override.地图坐标导航.attach.quests
          → 索引为 (chapter, map_quest_key)
      6. 写日志：哪些 quest_id 缺映射（未在翻译表 / 未在 quests/*.json 中）
    """
```

#### 5.3 新增 `agent/custom/weekly_mission_action.py`

```python
@AgentServer.custom_action("solve_and_run_weekly_missions")
class SolveAndRunWeeklyMissions(CustomAction):
    def run(self, context, argv):
        a = context.get_node_data("周常任务自动执行")["attach"]

        # 构建索引
        chapter_overrides = load_chapter_overrides()  # 解析 chapter.json
        quest_overrides   = load_quest_overrides()    # 解析 quests/*.json
        id_to_loc         = build_quest_id_index()    # quest_id → (chapter, map_key)

        # 求解
        missions = get_current_missions(a["region"])
        quests   = get_free_quests(a["region"], a["progress"])
        plan     = solve(quests, missions).plan

        # 逐关执行
        for quest_id, count in plan.items():
            loc = id_to_loc.get(quest_id)
            if not loc:
                mfaalog.warning(f"无映射: {quest_id}, 跳过")
                continue
            chapter, map_key = loc

            # 合并三层 override：章节 template + 关卡 + BBC 参数
            full_override = {
                **chapter_overrides[chapter],
                **find_quest_override(quest_overrides, chapter, map_key),
                "执行BBC任务": {
                    "attach": {
                        "bbc_team_config":         a["bbc_team_config"],
                        "run_count":               count,
                        "apple_type":              a["apple_type"],
                        "battle_type":             a["battle_type"],
                        "support_order_mismatch":  a["support_order_mismatch"],
                        "team_config_error":       a["team_config_error"],
                        "connect":                 a["connect"],
                        "mumu_path":               a["mumu_path"],
                        # ... 其他模拟器参数
                    }
                }
            }
            context.override_pipeline(full_override)

            # 跑一次完整状态机 → BBC 内部刷 N 次 → 战斗完成兜底退出
            ok = context.run_task("通用战斗调度")
            if not ok:
                return CustomAction.RunResult(success=False)

        return CustomAction.RunResult(success=True)
```

## 6. 关键边界 / 待运行时验证

1. **BBC 用同一 config 打不同 quest 是否可行**：依赖 MaaFgo 已经导航到目标关卡选择界面后，BBC 出击命令是否忽略 config 中绑定的 quest_id 直接出击。**第一阶段按"BBC 不在意 quest 字段"假设跑**，失败时改方案：为每个 quest 复制并改写 config 中的 quest 字段（仍用同一队伍编成）。
2. **`run_task("通用战斗调度")` 退出条件**：BBC 打完 N 次回到主界面后，期望走 `战斗完成信息` 兜底分支退出。需实测验证不会卡在某个 JumpBack。
3. **求解器只支持 CN**：`load_quest_enemies("JP")` 会 FileNotFoundError。MVP 只支持 CN，option 默认 CN。
4. **失败处理**：单关失败时**整体中止**（避免继续浪费苹果），并通过 GUI 输出失败信息（已有 `bbc弹窗信息输出` 节点机制）。
5. **章节/关卡名匹配**：`quest_enemies_CN.json` 的 `name` 与 `quests/{章节}.json` 中关卡 case 的 `name` 必须能对齐。第一阶段加 dry_run 模式离线验证全命中率。
6. **修炼场（迦勒底之门 War 1003）**：不在主线大地图上，导航方式不同。MVP 阶段先排除（不让求解器推荐），或者用单独的导航子图。

## 7. 验证策略

依赖 BBC 实机运行，纯单测覆盖不到全链路。分阶段：

1. **离线 dry_run**（不接 BBC）：custom_action 加 `dry_run` attach 开关，跑求解 + 映射 + 日志输出"理论会执行的 (chapter, quest_name, count) 序列"，确认全部映射命中。
2. **单关试跑**：硬编码 `plan = {某 quest_id: 1}`，跑通"导航 → BBC 1 次 → 战斗完成兜底退出"。
3. **多关串跑**：完整执行当周方案，观察章节切换是否成功。
4. **CI**：`npx @nekosu/maa-tools check` + `python tools/validate_schema.py` 验证 pipeline JSON。
